# Блок 4. Тестирование API

## Описание задачи
Спроектировать тестирование REST API для контроллера `/api/v2/orders` (управляющего жизненным циклом заказа) на основе индустриальных стандартов (Google REST API Guide, Microsoft REST API Guidelines).

Требуется:
- Составить чек-лист, проверяющий корректное поведение контроллера.
- Явно указать ожидаемые HTTP статус-коды при:
  - Успешном выполнении
  - Ошибках валидации
  - Отсутствии прав доступа (401/403)
  - Работе с несуществующим ID
  - Падении БД (500)

---

## Ход решения и матрица HTTP статус-кодов

Индустриальные гайдлайны от Google и Microsoft сходятся в том, что HTTP статус-коды должны точно отражать семантику ответа. REST-ресурсы должны быть идемпотентными там, где это необходимо, и возвращать понятные сообщения об ошибках в едином формате (например, RFC 7807 Problem Details).

### Матрица ожидаемых HTTP статус-кодов для `/api/v2/orders`

| Сценарий | HTTP статус-код | Описание |
| :--- | :--- | :--- |
| **Успешное получение списка** | `200 OK` | Возвращается массив объектов. При отсутствии записей — пустой массив `[]` (не NULL, не 404). |
| **Успешное получение по ID** | `200 OK` | Возвращается объект заказа. |
| **Успешное создание заказа** | `201 Created` | Возвращается созданный объект + заголовок `Location` со ссылкой на новый ресурс. |
| **Успешное обновление** | `200 OK` / `204 No Content`| `200 OK` если возвращается обновленная модель (Read-Model), `204 No Content` если тело ответа пустое. |
| **Успешное удаление** | `204 No Content` | Ресурс успешно удален (или логически помечен как удаленный). |
| **Повторное удаление / Запрос удаленного**| `410 Gone` | Если запись удалена логически (мягкое удаление), возвращается `410 Gone`. Если физически — `404 Not Found`. |
| **Ошибка валидации контракта** | `400 Bad Request` | Неверный формат JSON, некорректные типы данных. |
| **Бизнес-ошибки валидации** | `422 Unprocessable Entity`| Формат верный, но нарушены бизнес-правила (например, вес заказа больше грузоподъемности машины). |
| **Пользователь не авторизован** | `401 Unauthorized` | Отсутствует или невалиден JWT-токен / API-ключ в заголовке `Authorization`. |
| **Недостаточно прав (Роли)** | `403 Forbidden` | Токен валиден, но у пользователя (например, роли Водитель) нет прав на создание/удаление заказов. |
| **Обращение к несуществующему ID** | `404 Not Found` | Ресурс с указанным UUID не найден в БД. |
| **Конфликт ресурсов / Дубликат ID** | `409 Conflict` | Попытка создать заказ с UUID, который уже существует в системе (для `POST /orders/{id:uuid}`). |
| **Падение БД / Системный сбой** | `500 Internal Server Error` | Ошибка подключения к СУБД, падение пула соединений. Ошибка скрывает детали реализации от клиента. |

## Чек-лист проверки API контроллера `/api/v2/orders`

### 1. Получение списка заказов (`GET /api/v2/orders`)
*   [ ] **Happy Path:** Отправить GET-запрос авторизованным пользователем. Ожидается `200 OK` и JSON-массив заказов.
*   [ ] **Проверка пустого списка:** Очистить базу заказов, сделать запрос. Ожидается `200 OK` с телом `[]` (пустой массив).
*   [ ] **Фильтрация удаленных записей:** Убедиться, что логически удаленные заказы (со статусом deleted) не выводятся в общем списке.
*   [ ] **Пагинация и сортировка:** Проверить параметры `limit`, `offset` и `sort`. Пример: `/api/v2/orders?limit=10&offset=20&sort=-createdAt`. Ожидается `200 OK` с метаданными пагинации.
*   [ ] **Безопасность:** Сделать запрос без заголовка `Authorization`. Ожидается `401 Unauthorized`.

### 2. Получение заказа по ID (`GET /api/v2/orders/{id:uuid}`)
*   [ ] **Happy Path:** Запрос существующего UUID. Ожидается `200 OK` с полной схемой заказа.
*   [ ] **Неверный формат ID:** Запрос с некорректным форматом (например, `/orders/123-abc` вместо UUIDv4). Ожидается `400 Bad Request`.
*   [ ] **Несуществующий ID:** Запрос случайного UUID. Ожидается `404 Not Found`.
*   [ ] **Логически удаленный ID:** Запрос ID, который был ранее удален через `DELETE` (мягкое удаление). Ожидается `410 Gone`.
*   [ ] **Падение СУБД:** Имитировать обрыв связи с СУБД при поиске заказа. Ожидается `500 Internal Server Error` с безопасным текстом ошибки.

### 3. Создание заказа (`POST /api/v2/orders`)
*   [ ] **Happy Path (Классический POST):** POST-запрос на `/api/v2/orders` с валидным телом. Ожидается `201 Created`, объект заказа в теле ответа, и заголовок `Location: /api/v2/orders/{new_uuid}`.
*   [ ] **POST с принудительным ID (Идемпотентный):** Отправка POST на `/api/v2/orders/{id:uuid}`.
    *   Если ID новый -> `201 Created`.
    *   Если ID уже существует в БД -> `409 Conflict` (предотвращение дублирования транзакций).
*   [ ] **Ошибка схемы данных:** Отправить невалидный JSON или нарушить типы полей. Ожидается `400 Bad Request`.
*   [ ] **Ошибка бизнес-валидации:** Отправить дату погрузки в прошлом. Ожидается `422 Unprocessable Entity` с описанием нарушенного правила.

### 4. Обновление заказа (`PUT /api/v2/orders/{id:uuid}`)
*   [ ] **Happy Path (Полное обновление):** Отправить валидное измененное тело заказа.
    *   Если API возвращает обновленный объект -> `200 OK`.
    *   Если API не возвращает тело -> `204 No Content`.
*   [ ] **Частичное обновление (Нарушение семантики PUT):** Отправить PATCH-запрос вместо PUT для изменения одного поля. PUT-запрос с отсутствующими обязательными полями должен приводить к `400 Bad Request` или затирать опущенные поля значениями по умолчанию.
*   [ ] **Обновление несуществующего заказа:** Попытка применить PUT к случайному UUID.
    *   Если система поддерживает "Создание через PUT" -> `201 Created`.
    *   Если система требует только обновление -> `404 Not Found`.

### 5. Удаление заказа (`DELETE /api/v2/orders/{id:uuid}`)
*   [ ] **Happy Path (Первичное удаление):** Отправить DELETE-запрос на существующий UUID. Ожидается `204 No Content` (тело ответа пустое).
*   [ ] **Повторное удаление (Идемпотентность):** Повторить тот же DELETE-запрос к тому же UUID.
    *   *Вариант А:* Система возвращает `204 No Content` (так как цель — удаление — уже достигнута).
    *   *Вариант Б (Рекомендуемый для логического удаления):* Система возвращает `410 Gone` (ресурс удален окончательно).
*   [ ] **Удаление несуществующего:** Удаление UUID, которого никогда не было в системе. Ожидается `404 Not Found`.

### 6. Запрос заголовков (`HEAD /api/v2/orders/{id:uuid}`)
*   [ ] **Happy Path:** Запрос метаданных существующего заказа. Ожидается `204 No Content` (или `200 OK` без тела), но заголовки ответа (например, `Content-Length`, `ETag`) должны быть идентичны заголовкам обычного `GET` запроса.
