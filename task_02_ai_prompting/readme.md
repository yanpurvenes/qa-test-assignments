# Блок 2. Проектирование тестов (AI Prompting & Review)

## Описание задачи
1.  Сгенерировать с помощью LLM чек-лист для тестирования формы создания заявки на перевозку для «УЛ Лайт».
2.  Поля формы:
    *   Город отправления (автокомплит/выпадающий список из БД).
    *   Город назначения (автокомплит/выпадающий список из БД).
    *   Вес груза (числовое поле, в кг, максимум 20 000 кг).
    *   Объем груза (числовое поле, в м³, максимум 82 м³).
    *   Дата погрузки (датапикер, нельзя выбрать прошедшую дату).
    *   Кнопка «Рассчитать стоимость» (отправляет POST-запрос к API, возвращает цену в рублях или ошибку, если маршрут не найден).
3.  Опубликовать:
    *   Промпт.
    *   Сырой ответ ИИ.
    *   Экспертное ревью ответа ИИ (критический анализ, ошибки ИИ, упущенные бизнес-требования логистики, UX, интеграции, избыточные тесты).
    *   Файл с граничными условиями и обоснованием.

---

## 1. Наш Промпт для ИИ

```text
Роль: Senior QA Engineer в логистической компании.
Задача: Составь чек-лист (позитивные, негативные сценарии и граничные значения) для тестирования MVP формы создания заявки на перевозку в веб-приложении "УЛ Лайт".

Спецификация формы:
1. Поле "Город отправления": автокомплит/выпадающий список из БД городов.
2. Поле "Город назначения": автокомплит/выпадающий список из БД городов.
3. Поле "Вес груза": числовое, кг, максимум 20 000 кг.
4. Поле "Объем груза": числовое, кубические метры, максимум 82 м³.
5. Поле "Дата погрузки": датапикер, выбор прошедших дат заблокирован.
6. Кнопка "Рассчитать стоимость": отправляет POST на API, возвращает цену или ошибку "Маршрут не найден".

Выдай структурированный чек-лист в формате Markdown.
```

## 2. Сырой ответ ИИ (Raw AI Output)

```markdown
### Чек-лист тестирования формы «УЛ Лайт»

#### Позитивные сценарии:
1. [ ] Успешный расчет стоимости при заполнении всех полей валидными данными (Город отправления: Москва, Город назначения: Санкт-Петербург, Вес: 1000 кг, Объем: 10 м³, Дата: Сегодня + 1 день).
2. [ ] Выбор городов из выпадающего списка автокомплита при вводе первых 3-х букв.
3. [ ] Расчет стоимости при вводе минимального веса (1 кг) и минимального объема (1 м³).
4. [ ] Расчет стоимости при вводе максимального веса (20 000 кг) и максимального объема (82 м³).
5. [ ] Выбор сегодняшней даты в датапикере.
6. [ ] Выбор даты в далеком будущем (например, через год).

#### Негативные сценарии:
7. [ ] Попытка ввода отрицательного веса (-500 кг) — ожидается ошибка валидации.
8. [ ] Попытка ввода отрицательного объема (-5 м³) — ожидается ошибка валидации.
9. [ ] Попытка ввода веса больше максимума (20 001 кг) — ожидается блокировка ввода или ошибка.
10. [ ] Попытка ввода объема больше максимума (83 м³) — ожидается блокировка ввода или ошибка.
11. [ ] Ввод несуществующего города в автокомплит (например, "ВымышленныеВасюки") — ожидается пустой результат в поиске и ошибка «Город не найден».
12. [ ] Совпадение города отправления и города назначения — ожидается ошибка валидации формы.
13. [ ] Попытка отправить пустую форму — кнопка «Рассчитать стоимость» неактивна или выводит ошибки у пустых полей.
14. [ ] Попытка ввести прошедшую дату в поле даты вручную (если поле доступно для ввода) — ожидается ошибка.

#### Граничные значения:
15. [ ] Вес груза: 0 кг, 1 кг, 19 999 кг, 20 000 кг, 20 001 кг.
16. [ ] Объем груза: 0 м³, 1 м³, 81 м³, 82 м³, 83 м³.
```

## 3. Экспертное ревью ответа ИИ

Сырой ответ ИИ является типичным примером "поверхностного" тестирования UI. Он упускает специфику логистики, особенности клиент-серверного взаимодействия и UX.

### Критические ошибки и логистические упущения ИИ:

1.  **Проблема соотношения Веса и Объема (Логистическая несовместимость):**
    *   *Что упустил ИИ:* В реальности груз весом 20 000 кг (20 тонн) физически не может занимать объем 0.1 м³ (это плотность в сотни раз выше золота), а груз объемом 82 м³ (полная фура) не может весить 1 кг (если это не воздух).
    *   *Как надо:* На бэкенде или фронтенде должна быть проверка на корректность плотности груза (соотношение веса к объему) или расчет удельного объема. ИИ этого не учел.
2.  **Специфика автокомплита городов (Сетевой уровень и производительность):**
    *   *Что упустил ИИ:* Поля городов подгружаются из БД. ИИ проверил только "ввод 3 букв".
    *   *Как надо:* 
        *   Проверить задержку отправки запросов к БД при вводе (Debounce rate, например, 300 мс), чтобы не слать запросы на каждую букву и не класть базу данных.
        *   Кейс с городами-тезками (например, Киров в Кировской области и Киров в Калужской области). Пользователь должен видеть область/район в списке автокомплита для однозначной идентификации.
3.  **Часовые пояса в датапикере (TimeZone Issue):**
    *   *Что упустил ИИ:* Выбор "сегодняшней даты".
    *   *Как надо:* Если пользователь находится во Владивостоке (UTC+10), а сервер расчета цены в Москве (UTC+3), то в 23:00 во Владивостоке в Москве уже другое число. Датапикер должен рассчитывать доступность дат относительно локального времени погрузки, а не системного времени сервера, чтобы не заблокировать валидную дату.
4.  **Сбои интеграции API (HTTP Status Codes):**
    *   *Что упустил ИИ:* Нажатие кнопки отправляет POST-запрос.
    *   *Как надо:* Описать обработку ошибок API: что делать при таймауте ответа расчета (более 5 секунд), при HTTP 502/503 (сервер расчета тарифов недоступен).
5.  **Избыточные или нереалистичные кейсы ИИ:**
    *   *Кейс 16 (0 м³):* Нулевой объем — это физически невозможный груз. Эта проверка избыточна, так как нижняя граница должна проверяться на уровне 0.01 м³ (или минимальный шаг системы).

## 4. Граничные условия для тестирования (Спецификация)

Для MVP формы «УЛ Лайт» критичны следующие граничные точки:

| Параметр | Граничная точка | Тип проверки | Логическое обоснование |
| :--- | :--- | :--- | :--- |
| **Вес груза** | **0.00 кг** | Негативный | Вес груза должен быть строго положительным. |
| | **0.01 кг** (10 г) | Позитивный | Минимально возможный регистрируемый вес (мелкая посылка). |
| | **19 999.99 кг** | Позитивный | Максимум минус минимальный шаг. |
| | **20 000.00 кг** | Позитивный | Верхняя граница грузоподъемности стандартной еврофуры (20 т). |
| | **20 000.01 кг** | Негативный | Перегруз. Риск штрафов для перевозчика, форма должна блокировать отправку. |
| **Объем груза** | **0.00 м³** | Негативный | Объем должен быть строго больше нуля. |
| | **0.01 м³** | Позитивный | Минимальный регистрируемый объем. |
| | **81.99 м³** | Позитивный | Максимум минус шаг. |
| | **82.00 м³** | Позитивный | Стандартный максимальный объем полуприцепа (82 куба). |
| | **82.01 м³** | Негативный | Превышение физического объема кузова стандартной фуры. |
| **Дата погрузки** | **Текущий день (Сегодня)** | Позитивный | Погрузка "день в день" — критична для срочных заказов. |
| | **Прошедший день (Вчера)** | Негативный | Запрещено правилами логистики (нельзя заказать машину на вчера). |
| **Длина строк городов**| **1 символ** | Негативный | Автокомплит не должен перегружать сервер запросами при вводе одной буквы. |
| | **3 символа** | Позитивный | Стандартный порог (минимальная длина префикса для старта поиска). |
