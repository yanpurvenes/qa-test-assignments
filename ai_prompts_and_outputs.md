## Промпты и сырые ответы ИИ (Блок 2: AI Prompting)

Для Блока 2 («Проектирование тестов») ИИ выполнял роль генератора и одновременно критика. Ниже приведены детали этого процесса.

### Шаг A: Промпт генерации (вводные данные для генерации чек-листа)
```text
Роль: Senior QA Engineer в логистической компании.
Задача: Составь чек-лист (позитивные, негативные сценарии и граничные значения) для тестирования MVP формы создания заявки на перевозку в веб-приложении "УЛ Лайт".

Спецификация формы:
1. Поле "Город отправления": автокомплит/выпадающий список из БД городов.
2. Поле "Город назначения": автокомплит/выпадающий список из БД городов.
3. Поле "Вес груза": числовое, кг, максимум 20 000 кг.
4. Поле "Объем груза": числовое, кубические метры, максимум 82 м³.
5. Поле "Дата погрузки": датапикер, выбор прошедших дат заблокирован.
6. Кнопка "Рассчитать стоимость": отправляет POST на API, возвращает цену или ошибку "Маршрут не найден".

Выдай структурированный чек-лист в формате Markdown.
```

### Шаг B: Сырой ответ ИИ (зафиксированный в `task_02_ai_prompting/readme.md`)
ИИ сгенерировал базовый список проверок (ввод отрицательного веса, совпадение городов, граничные точки 0, 1, 20000, 20001).

### Шаг C: Критический разбор ответов (Архитектурный кейс)
В процессе ревью результатов работы ИИ были выявлены следующие слабые места, которые реальный Senior QA исправил бы на практике:
*   *Игнорирование плотности груза (Вес/Объем):* Нейросеть не сопоставила вес и объем. Например, фура объемом 82 м³ физически не может вместить 20 тонн ваты (объем превысит норму), а 20 тонн металла займут не более пары кубометров. Нужна бизнес-проверка плотности.
*   *Отсутствие проверок часового пояса (Timezones):* Валидация даты «сегодня» относительно часового пояса отправления (например, Владивосток/Москва) — классическая логистическая проблема, которую ИИ пропустил.
*   *Нагрузка на сеть при автокомплите:* ИИ не предложил проверить задержку ввода (`Debounce`) для предотвращения ddos-эффекта на базу данных городов.

## Архитектурные решения по интеграционным блокам (Блоки 4 и 5)

*   **REST API (Блок 4):** Чек-лист спроектирован с упором на стандарты Google REST API и Microsoft Guidelines. Явно разграничены ошибки `400 Bad Request` (ошибки контракта JSON) и `422 Unprocessable Entity` (бизнес-ошибки логики, например, несовместимость маршрутов).
*   **1С-Движок (Блок 5):**
    *   При ревью функции `РассчитатьСтоимость` было выявлено критическое использование физической таблицы вместо виртуальной таблицы `СрезПоследних`. Это классическая ошибка 1С-разработчиков, приводящая к блокировкам СУБД.
    *   Исключена логическая ошибка с оператором `МЕЖДУ`, которая приводила к наложению интервалов веса на стыке (вес = 10). Границы веса переписаны на строгие неравенства `> ВесОт И <= ВесДо`.
